# v1 — Browser MuJoCo + Unitree Locomotion Policy (inference only)

Goal: a webpage that runs MuJoCo physics and a pre-trained Unitree locomotion policy entirely in-browser. No server-side compute, no streaming. Single robot, single task.

## Stack

| Layer        | Choice                          | Why                                                              |
| ------------ | ------------------------------- | ---------------------------------------------------------------- |
| Physics      | `@mujoco/mujoco` v3.8.0 (npm)   | Official DeepMind WASM bindings; supersedes zalo/forks           |
| Rendering    | Three.js                        | De-facto choice; reads MuJoCo geom state directly                |
| Policy       | `onnxruntime-web` (WebGPU/WASM) | Framework-agnostic; export from PyTorch via `torch.onnx.export`  |
| Build        | Vite + TypeScript               | Fast dev server; native ESM; trivial COOP/COEP config            |
| Concurrency  | Web Workers                     | Physics + policy off the main thread to keep render at 60 fps    |

## Architecture

```
Worker A (physics)   @mujoco/mujoco/mt   ─┐
                                          ├─► SharedArrayBuffer (state, action)
Worker B (policy)    onnxruntime-web     ─┘
Main thread          Three.js render loop, UI, asset loading
```

Rates:
- Physics: 500 Hz (matches MuJoCo default `dt=0.002`)
- Policy: 50 Hz (typical Unitree control rate; 1 inference per 10 sim steps)
- Render: vsync (~60 Hz)

## Robot + Task

- **Robot**: Unitree Go2 (quadruped — simpler than humanoid for v1).
- **Asset**: `unitreerobotics/unitree_mujoco` MJCF + meshes (Apache-2.0).
- **Policy**: walking policy exported to ONNX. For v1, use a public checkpoint from `unitree_rl_gym` or train a minimal one in IsaacLab/MJX offline. Either way the browser only loads `policy.onnx`.

## Observation / action contract (must match training exactly)

This is where browser ports fail silently. Lock the contract before wiring it:
- Obs vector: base ang vel (3), projected gravity (3), commanded velocity (3), joint pos − default (12), joint vel (12), last action (12) = **45 dims** (exact layout depends on the source repo).
- Action: target joint positions, scaled and added to defaults, then PD-controlled by MuJoCo `position` actuators.
- Joint ordering between training MJCF and browser MJCF must be identical.
- Plan: dump 100 obs/action pairs from the training env, replay them through the browser MuJoCo, assert agreement before integrating the policy.

## Hosting

Multi-threaded WASM build needs cross-origin isolation:
```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```
Vite dev server: set in `vite.config.ts`. Production: any host that allows custom headers (Cloudflare Pages, Vercel, Netlify all fine).

## Build steps

1. Vite + TS scaffold, install `@mujoco/mujoco`, `three`, `onnxruntime-web`.
2. Configure COOP/COEP headers in Vite dev server.
3. **Milestone 1** — Load `humanoid.xml` (ships with the package), step physics, log qpos. No rendering.
4. **Milestone 2** — Three.js scene; draw MuJoCo geoms (boxes, spheres, capsules, planes, meshes) by syncing per-frame from `mjData.xpos` / `xmat`.
5. **Milestone 3** — Load Go2 MJCF + meshes; verify it stands and falls under gravity.
6. **Milestone 4** — Move physics into a Web Worker; SharedArrayBuffer for state.
7. **Milestone 5** — Load `policy.onnx` in a second Worker; build obs vector; write actions back; close the loop.
8. **Milestone 6** — Keyboard/gamepad input → commanded velocity; basic UI (reset, pause, step).

## Out of scope for v1

- Multiple robots / scenes
- Training in browser
- Domain randomization, sim-to-real noise
- Mobile / touch controls
- Multi-task or language-conditioned policies

## Risks

1. **Obs parity bugs** — biggest source of "policy works in Python, falls over in browser." Mitigation: parity test in step 5 before plugging in the policy.
2. **Mesh asset size** — Go2 meshes can be tens of MB; will need to host them as static files and stream.
3. **WASM perf** — single Go2 at 500 Hz should be comfortable on a laptop; verify early.
4. **Threading flakiness on Safari** — multi-threaded build path needs a single-threaded fallback.
